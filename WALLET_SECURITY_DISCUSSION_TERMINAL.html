The big finding: WebAuthn PRF extension is the real answer.

In plain English: when you use a passkey (fingerprint/face scan), the PRF extension can produce a secret encryption key that's tied to your biometrics. That key encrypts your BSV private key. The encrypted key sits in storage — useless without your fingerprint. The encryption key is never stored anywhere.
What works today:
  • Chrome, Safari, Edge — all support PRF
  • iCloud Keychain and Google Password Manager sync passkeys AND preserve PRF output across devices
  • Firefox — unreliable, needs a fallback
The catch: If you lose your passkey AND your backup, the BSV key is gone forever. So you still need a backup export option.

All 3 agents are back. Here's the complete, simple picture:
The Simple Version
The problem: BSV uses a curve called secp256k1. Almost no hardware chip supports it — not Apple's Secure Enclave, not TPMs, not passkeys, not YubiKeys. So hardware can't directly be your BSV key.

The solution: Hardware can guard your BSV key instead.
Your Options, Ranked
1. Passkey-Wrapped Key (Best option today)
How it works: Your BSV key is encrypted. To decrypt it, you scan your fingerprint or face. The passkey (backed by your device's secure chip) produces an encryption key that unlocks your BSV key. No fingerprint = no key.
AspectDetails
UXNearly invisible — fingerprint/face scan takes <1 second
SecurityStolen localStorage = useless ciphertext. Attacker needs your face + device
Works onChrome, Safari, Edge. Syncs via iCloud/Google across devices
WeaknessBSV key briefly exists in browser memory during signing
Progressive?Yes — start in localStorage now, add passkey wrapping later
EffortMedium — WebAuthn PRF extension + encrypt/decrypt logic
Both agents say this is the move.
2. Server-Side HSM (If budget allows)
Your server holds keys in tamper-proof hardware (AWS CloudHSM supports secp256k1). User authenticates with passkey, server signs on their behalf. Key never touches a browser.
  • Zero client-side friction
  • ~$1,100/month for AWS CloudHSM
  • Makes you custodial (you hold everyone's keys)
3. Threshold Signing (Future)
Key is split in two: half in user's browser (passkey-protected), half on your server (HSM). Neither half can sign alone. Both are needed. Best of both worlds — but cutting-edge crypto, hard to implement correctly.
4. Hardware Wallet (Ledger/Trezor)
Supports secp256k1, but terrible UX — plug in USB, enter PIN, confirm on tiny screen for every post. BSV support is shaky (Ledger removed the BSV app). Not realistic for your use case.
5. Secure Enclave / TPM / YubiKey directly
Dead end. None support secp256k1. Can only be used indirectly via passkeys (Option 1).

When real money is involved, the safe answer is: generate a NEW key after upgrading.
The flow would be:
1. User upgrades to PIN → gets a fresh keypair (new key, new address)
2. Old key is retired, old posts still show under old name

This means their identity changes. But you can link them: the old key signs a message saying "I'm moving to this new address" — on-chain proof of migration. Old posts still belong to them, new funds go to the clean key.

BSV Agent Findings
Self-Funded Posting — How It Actually Works
1. Check UTXOs via WhatsOnChain API: GET /address/{address}/unspent
2. Build transaction with OP_RETURN (post content in MAP format) + change output back to user
3. Sign with user's key client-side
4. Broadcast
5. If no UTXOs → server funds it instead
Critical detail: You MUST create a change output or the user loses their remaining balance. The SDK handles this.
Cost: ~1 satoshi per post. 10,000 satoshis ($0.001) = thousands of posts.
Daily Limits — On-Chain Won't Work
Per-address limiting is not effective on-chain because generating a new address is free. A spammer just makes new identities.
What works instead:
  • Server-side enforcement (pubkey + IP + session token)
  • Proof-of-work for free posts (client computes a hash, raises the bar for spam)
  • BSV itself is anti-spam — when users self-fund, every post costs money. The problem is only the free tier.
Suggested model: 5 free posts/day (server-funded). Hit the limit → show QR to fund. Once funded, no server limit — the chain's fee market handles it.
Migration Message — MAP + AIP Protocol
The on-chain migration would be an OP_RETURN signed by the old key:
MAP SET app "bfn" type "identity-migration" old_address "..." new_address "..." + AIP signature proving the old key authorized it
Server should be the arbiter for migration (not chain-only), since key theft is a known risk. Options: timelock, sequence numbers, or multi-factor confirmation.
Things to Add
  • Address column in posts table — derivable from pubkey but makes querying way easier
  • Nonce/timestamp in signed payloads — prevents replay attacks
  • Switch to BSM format (Bitcoin Signed Message) instead of raw ECDSA — compatible with standard BSV tools
  • Plan for Yours Wallet integration — power users will want to connect existing wallets via @1sat/connect
  • UTXO consolidation — when users accumulate many tiny UTXOs from micro-revenue, batch them periodically
