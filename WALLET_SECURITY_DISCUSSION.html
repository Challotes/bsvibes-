🔐 Wallet Security Discussion

Exploring how Build From Nothing protects user keys — and the upgrade path to hardware-level security

Current State: localStorage

How it works now: When a user visits the site, a BSV keypair is generated in-browser and stored in localStorage as a WIF (Wallet Import Format) string. The key is stored in plaintext.

Risk: Anything that can read localStorage — an XSS attack, a malicious browser extension, someone with DevTools access — can grab the raw private key.

This is acceptable for the initial "idea board" phase where no real money is at stake. But it must be upgraded before any BSV flows to user addresses.

The Upgrade Path

Stage What Protects the Key Recovery Method
Now Nothing (raw key in localStorage) None — if browser clears, key is lost
Stage 2: PIN/Passphrase AES-256 encryption with user passphrase Backup file download
Stage 3: Passkey Wrapping AES-256 via WebAuthn PRF (biometric) Backup file + passkey sync
Stage 4: Server HSM Tamper-proof hardware (AWS CloudHSM) Server-managed recovery

Passkey-Wrapped Keys (Recommended Path)

The big finding: WebAuthn PRF extension is the answer. When you use a passkey (fingerprint/face scan), the PRF extension produces a secret encryption key tied to your biometrics. That key encrypts your BSV private key. The encrypted key sits in storage — useless without your fingerprint. The encryption key is never stored anywhere.

Browser Support

  • Chrome, Safari, Edge — all support PRF
  • iCloud Keychain & Google Password Manager — sync passkeys AND preserve PRF output across devices
  • Firefox — unreliable, needs a passphrase fallback

Why Not Direct Hardware?

BSV uses secp256k1 curve. Almost no hardware chip supports it — not Apple's Secure Enclave, not TPMs, not passkeys, not YubiKeys. So hardware can't directly be your BSV key. But hardware can guard your BSV key via passkey wrapping.

The Natural Upgrade Flow

The user self-selects into security:

  1. User clicks "save your key" (becomes "keep your name")
  2. They download a backup file (encrypted with passphrase)
  3. Right after: "Want to protect your account with Face ID?"
  4. They tap yes → key gets passkey-wrapped

Users who don't care never see any of this. Zero friction stays zero friction.

The 5-Minute Window Problem

Real risk: If the key sat in plain localStorage for any amount of time — even 5 minutes — you have to assume it could have been copied. An XSS attack, a browser extension, or malware could have silently read it.

Solution: When real money is involved, generate a NEW key at upgrade time. The old key signs a migration message on-chain, and the new key is encrypted from birth.

When to Rotate Keys

  • Early phase (now): Don't rotate. Risk is near-zero for an idea board with no funds
  • Revenue phase: Everyone must upgrade. Fresh key, PIN/passkey-protected from birth. On-chain migration message links old → new address

Self-Funded Posting

When a user's address has BSV, the app can spend from their balance instead of the server paying:

Now: User clicks Post → app signs for proof → server pays the fee → on-chain Later: User clicks Post → app signs for proof AND payment in one tx → on-chain

Same button, same UX. If their balance runs out, the server picks up the tab again. Seamless.

Posts Today What Happens

Hit limit, no BSV "Daily limit reached. Fund your address to continue." + QR code

Agent Consensus

All 3 agents agreed:

  1. Fix server-side signature verification first — the identity system is decorative without it
  2. Passkey-wrapped keys are the best path to hardware-level security
  3. PIN → Passkey is a valid upgrade path — each stage is strictly better
  4. Self-funded posting can be 100% seamless
  5. Key rotation at revenue time is the clean answer to the 5-minute window
  6. Rename everything — users care about "keeping their name", not "saving their key"
